\TFMSection{Análisis de problema y caso de uso}
El caso de uso principal del sistema es la generación asistida de una propuesta de viaje personalizada, desde la carga inicial de documentación hasta la entrega de una propuesta comercial revisable por un profesional. El flujo completo se compone de las siguientes fases:

\begin{enumerate}
\item \textbf{Carga de documentación.} Un documento de un proveedor turístico (tarifario, catálogo de actividades, ficha de alojamiento, etc.) se sube al sistema en formato PDF.
\item \textbf{Procesamiento automático.} El sistema convierte el documento a imágenes, extrae su contenido a Markdown, normaliza su estructura, extrae metadatos y lo indexa en la base de conocimiento, quedando disponible para su consulta sin intervención manual.
\item \textbf{Inicio de una conversación.} El profesional de la agencia inicia una conversación en lenguaje natural describiendo el viaje que necesita organizar.
\item \textbf{Recopilación de requisitos del viaje.} El sistema identifica en la conversación los datos necesarios para continuar (cliente, origen, destino, fechas e intereses) y solicita aclaraciones si falta información relevante.
\item \textbf{Consulta de información turística.} Una vez completos los requisitos, el sistema recupera de la base documental información y actividades relacionadas con el destino y los intereses indicados.
\item \textbf{Consulta de precios.} El sistema obtiene opciones de vuelo y alojamiento, combinando fuentes externas en tiempo real con información de proveedores disponible en la base documental, y presenta varias opciones al profesional para que seleccione la más adecuada.
\item \textbf{Elaboración del itinerario.} A partir de la información recuperada, los precios seleccionados y las preferencias expresadas en la conversación, el sistema genera un itinerario día a día.
\item \textbf{Generación de la propuesta comercial.} El sistema genera una imagen de portada asociada al destino y compone una propuesta comercial en formato editable, que el profesional revisa antes de aprobarla, corregirla con comentarios, descartarla o guardarla como borrador.
\end{enumerate}

Este caso de uso pone de manifiesto dos necesidades que condicionan el diseño: disponer de información turística siempre actualizada y correctamente recuperada de documentación heterogénea, y mantener al profesional en el centro de la decisión, de forma que el sistema proponga y confirme en lugar de automatizar decisiones con impacto comercial directo.

\TFMSection{Requisitos Funcionales}
A partir del análisis anterior, se han definido los siguientes requisitos funcionales:

\begin{itemize}
\item \textbf{RF1.} El sistema debe permitir la carga de documentación de proveedores turísticos y su conversión automática a un formato consultable.
\item \textbf{RF2.} El sistema debe extraer y normalizar el contenido de los documentos cargados (texto, tablas, precios, condiciones y metadatos asociados).
\item \textbf{RF3.} El sistema debe indexar semánticamente el contenido documental en una base de datos vectorial para permitir su recuperación mediante lenguaje natural.
\item \textbf{RF4.} El sistema debe recuperar información turística relevante para un destino y unos intereses concretos, fundamentando sus respuestas en la documentación disponible.
\item \textbf{RF5.} El sistema debe recopilar de forma conversacional los requisitos del viaje (cliente, origen, destino, fechas e intereses) y solicitar aclaraciones cuando la información sea insuficiente.
\item \textbf{RF6.} El sistema debe consultar precios de vuelos y alojamiento, combinando fuentes externas en tiempo real con información interna cuando la externa no esté disponible.
\item \textbf{RF7.} El sistema debe permitir al profesional seleccionar manualmente entre varias opciones de vuelo y alojamiento antes de continuar con la generación de la propuesta.
\item \textbf{RF8.} El sistema debe generar automáticamente un itinerario personalizado coherente con las fechas, el presupuesto y las preferencias expresadas.
\item \textbf{RF9.} El sistema debe generar una imagen de portada relacionada con el destino y las actividades seleccionadas.
\item \textbf{RF10.} El sistema debe generar una propuesta comercial en un formato editable, a partir de plantillas corporativas.
\item \textbf{RF11.} El sistema debe permitir al profesional aprobar, rechazar con comentarios, descartar o guardar como borrador la propuesta generada antes de su publicación.
\item \textbf{RF12.} El sistema debe permitir consultar el histórico de propuestas generadas y retomar una propuesta existente para continuar su edición.
\item \textbf{RF13.} El sistema debe autenticar a los usuarios antes de permitir el acceso a sus funcionalidades.
\end{itemize}

\TFMSection{Requisitos No Funcionales}
Además de los requisitos funcionales, se han considerado los siguientes requisitos no funcionales, derivados de las decisiones de diseño e implementación adoptadas:

\begin{itemize}
\item \textbf{RNF1 (Escalabilidad).} Los servicios de procesamiento y persistencia deben escalar automáticamente según la demanda, sin necesidad de aprovisionar capacidad de forma manual.
\item \textbf{RNF2 (Disponibilidad).} El pipeline de procesamiento documental debe ejecutarse de forma asíncrona y desacoplada del resto del sistema, de modo que un fallo puntual no bloquee el resto de funcionalidades.
\item \textbf{RNF3 (Seguridad).} El acceso a la aplicación debe estar protegido mediante autenticación, y las credenciales de los usuarios finales deben mantenerse separadas de las credenciales de servicio utilizadas internamente por el sistema.
\item \textbf{RNF4 (Resiliencia).} El sistema debe tolerar fallos transitorios de servicios externos (proveedores de modelos de lenguaje, fuentes de precios) mediante reintentos, y debe degradar su funcionamiento de forma controlada en lugar de interrumpir el proceso completo.
\item \textbf{RNF5 (Usabilidad).} La interacción con el sistema debe realizarse mediante una interfaz conversacional sencilla, con retroalimentación visible del progreso del proceso.
\item \textbf{RNF6 (Mantenibilidad).} La solución debe organizarse en componentes independientes con responsabilidades claramente delimitadas, de forma que puedan evolucionar o desplegarse por separado.
\item \textbf{RNF7 (Trazabilidad).} El sistema debe registrar los eventos relevantes de cada proceso y conservar la referencia a las fuentes documentales utilizadas en cada respuesta generada.
\item \textbf{RNF8 (Coste).} La selección de modelos y servicios debe tener en cuenta su coste de operación, priorizando alternativas eficientes sin comprometer la calidad del resultado.
\end{itemize}

\TFMSection{Arquitectura de la Aplicación}
La solución se ha diseñado como un conjunto de componentes independientes que se comunican entre sí, en lugar de como una única aplicación monolítica, de forma que cada uno pueda evolucionar, desplegarse y escalar por separado. La arquitectura se organiza en tres aplicaciones y un pipeline de procesamiento documental:

\begin{tfmbullets}
\item \textbf{Orquestador multiagente:} implementa el flujo de generación de propuestas mediante un grafo de agentes especializados construido con \textbf{LangGraph}.
\item \textbf{Interfaz de usuario:} aplicación web desarrollada en \textbf{Streamlit} que permite iniciar conversaciones, seleccionar opciones de precio, revisar propuestas y consultar el histórico generado.
\item \textbf{API backend:} servicio de persistencia unificado, implementado con \textbf{FastAPI}, que da acceso a los metadatos documentales, al estado de las propuestas y a los usuarios de la aplicación.
\item \textbf{Pipeline de procesamiento documental:} conjunto de funciones \textbf{AWS Lambda} encadenadas que transforman un documento de proveedor en información consultable por el sistema RAG.
\end{tfmbullets}

El orquestador y la interfaz se ejecutan dentro del mismo proceso, sin frontera HTTP entre ambos, lo que simplifica la interacción humana en el bucle al compartir el estado de la conversación y de la propuesta en el mismo ciclo de vida. El API backend, en cambio, se comunica con el resto de componentes mediante peticiones HTTP autenticadas, como capa de persistencia común para los metadatos documentales y el estado de las propuestas. El pipeline documental, por su parte, se ejecuta de forma completamente independiente y asíncrona: un documento puede cargarse y quedar disponible sin que el orquestador ni la interfaz intervengan.

\TFMSection{Diseño del pipeline documental}
El pipeline de procesamiento documental transforma un documento de proveedor en PDF en fragmentos indexados y disponibles para recuperación semántica, en tres etapas encadenadas:

\begin{enumerate}
    \item \textbf{Conversión de PDF a imágenes.} Una imagen por página para preservar la maquetación original (tablas, columnas, logotipos), inadecuada para extracción de texto convencional en tarifarios y catálogos.
    \item \textbf{Extracción, normalización y enriquecimiento.} Un modelo con capacidad de visión reconstruye el contenido en Markdown conservando su jerarquía, y se extraen metadatos estructurados (destino, proveedor, actividades, precios, condiciones), incorporando información de enlaces web cuando existen.
    \item \textbf{Fragmentación e indexación.} El contenido se divide en fragmentos de tamaño acotado, cada uno con su representación vectorial y metadatos, para su recuperación por similitud semántica y filtros exactos.
\end{enumerate}

Cada etapa recibe una entrada definida y un resultado reutilizable por la siguiente, lo que permite procesar de forma asíncrona, conservar resultados intermedios y reprocesar una etapa concreta cuando sea necesario. El resultado constituye la base documental del sistema RAG, usada principalmente por el agente de información y, de forma complementaria, por el de precios.

\TFMSection{Diseño del sistema multiagente}
El núcleo de la solución es un sistema multiagente que coordina las tareas necesarias para generar una propuesta de viaje: recopilar requisitos, recuperar información, consultar precios, elaborar el itinerario, generar una imagen y componer la propuesta final. Cada tarea se ha modelado como un agente especializado, coordinados mediante un grafo de estados con un estado compartido entre todos los nodos (Figura~\ref{fig:orquestrador}), cuya implementación se detalla en el apartado \ref{sec:impl-multiagente}.

El estado compartido almacena, entre otros elementos: los parámetros del viaje recopilados, el contexto cultural y las actividades recuperadas para el destino, las opciones de precio candidatas y la selección final, el contenido del itinerario, la ruta de la imagen de portada generada, el contenido de la propuesta comercial y su estado de revisión (borrador, publicada, rechazada o descartada).

Los agentes diseñados son los siguientes:

\begin{tfmbullets}
\item \textbf{Agente orquestador:} interpreta la conversación con el profesional, extrae los parámetros del viaje y determina si la información recopilada es suficiente para continuar o si es necesario solicitar una aclaración.
\item \textbf{Agente de información:} recupera, mediante el sistema RAG, información turística y actividades relacionadas con el destino y los intereses indicados.
\item \textbf{Agente de precios:} obtiene opciones de vuelo y alojamiento combinando fuentes externas en tiempo real con información de proveedores disponible en la base documental.
\item \textbf{Agente de itinerario:} elabora el itinerario día a día a partir de la información recuperada, los precios seleccionados y las preferencias expresadas en la conversación.
\item \textbf{Agente de imagen:} genera una imagen de portada asociada al destino y a las actividades propuestas.
\item \textbf{Agente de propuesta comercial:} compone la propuesta final en un formato editable a partir de plantillas corporativas, integrando el itinerario, la imagen de portada y los precios.
\end{tfmbullets}

El flujo no es estrictamente secuencial: la búsqueda de información y la consulta de precios se ejecutan en paralelo por ser independientes; igualmente, el itinerario y la imagen de portada se generan en paralelo una vez disponible la información necesaria, y ambos deben completarse antes de componer la propuesta final.

Un aspecto central del diseño es la incorporación de dos puntos de \emph{interacción humana en el bucle} (\textit{human-in-the-loop}), en los que el flujo se detiene de forma obligatoria hasta recibir una decisión del profesional:

\begin{enumerate}
\item \textbf{Selección de precios:} tras consultar las opciones de vuelo y alojamiento, el flujo se detiene para que el profesional seleccione manualmente la opción de cada categoría antes de continuar, tal como se muestra en la Figura~\ref{fig:manual-seleccion-precios}.
\item \textbf{Revisión final:} tras generar la propuesta comercial completa, el flujo se detiene para que el profesional la apruebe, la rechace indicando comentarios de corrección, la descarte o la guarde como borrador, tal como se muestra en la Figura~\ref{fig:manual-revision-flyer}.
\end{enumerate}

Este diseño responde a la necesidad, identificada en el análisis del problema, de mantener al profesional como responsable último de las decisiones con impacto comercial: el sistema genera una propuesta inicial revisable, no un sustituto de su criterio.

\TFMSection{Diseño de datos}
El diseño de datos de la solución combina tres tipos de almacenamiento, cada uno adecuado al tipo de información que gestiona:

\begin{tfmbullets}
\item \textbf{Almacenamiento documental (Amazon S3):} conserva los documentos originales, las imágenes generadas a partir de cada página y el contenido en Markdown resultante del procesamiento, así como las propuestas comerciales generadas en formato editable.
\item \textbf{Almacenamiento de estado (Amazon DynamoDB):} conserva los metadatos estructurados de cada documento procesado, el estado completo de cada propuesta en curso o finalizada (parámetros del viaje, selección de precios, itinerario, estado de revisión) y los usuarios de la aplicación.
\item \textbf{Base de datos vectorial (Amazon OpenSearch Serverless):} conserva los fragmentos de contenido documental indexados junto con su representación vectorial y sus metadatos de destino y actividades, permitiendo su recuperación semántica filtrada por destino.
\end{tfmbullets}

Esta separación permite gestionar cada dato con el modelo de acceso más adecuado: clave simple para metadatos y estado de propuestas, objetos de gran tamaño para documentos e imágenes, y búsqueda por similitud para el contenido indexado. Al ser una prueba de concepto con patrones de acceso simples (consulta por identificador, sin relaciones complejas), se ha priorizado un modelo de datos sencillo y de bajo mantenimiento frente a un esquema relacional.

\TFMSection{Selección Tecnológica e Infraestructura}
La selección tecnológica se ha realizado una vez definida la arquitectura anterior, para que las tecnologías respondan a las necesidades de cada componente y no al contrario.

\textbf{LangGraph} se eligió para el sistema multiagente por ofrecer de forma nativa las dos primitivas que exige el diseño: ejecución en paralelo de nodos independientes (\textit{fan-out}/\textit{fan-in}), necesaria para el agente de información y el de precios, y un mecanismo de interrupción (\texttt{interrupt\_before}) que detiene el grafo para una decisión humana y lo reanuda conservando el estado; implementar esto sobre una cadena secuencial habría exigido gestionar manualmente la sincronización y persistencia en cada punto de parada.

Para los modelos de lenguaje se combinan proveedores según el equilibrio coste/fiabilidad de cada tarea: \textbf{Google Gemini Flash-Lite} en el orquestador y el OCR documental (alto volumen, cuota gratuita elevada), y \textbf{OpenAI GPT-4o/GPT-5} en información, precios, itinerario e imagen (donde la fiabilidad de la salida estructurada es crítica). En todos los casos se usa la salida estructurada de LangChain sobre modelos pydantic, evitando el análisis manual de texto libre.

Para la infraestructura cloud se han elegido servicios gestionados de \textbf{AWS} de pago por uso: \textbf{S3} para documentos, imágenes y propuestas; \textbf{Lambda} para el pipeline documental y la API backend; \textbf{DynamoDB} bajo demanda para el estado de la aplicación, sin aprovisionar capacidad ni requerir un modelo relacional; y \textbf{OpenSearch Serverless} como base vectorial junto con \textbf{Bedrock} para los embeddings (\textit{Titan Text Embeddings v2}), manteniendo todo el pipeline de recuperación bajo el mismo proveedor y las mismas credenciales IAM. Esta combinación evita administrar servidores o dimensionar capacidad de antemano, adecuada para un volumen de uso variable y no predecible.

La API backend, en \textbf{FastAPI}, se expone sobre Lambda mediante \textbf{Mangum} como adaptador ASGI, ejecutando el mismo código en local (Uvicorn) y en producción (contenedor Lambda) sin mantener dos implementaciones. Para la interfaz se optó por \textbf{Streamlit}, que permite una interfaz conversacional funcional con poco código, priorizando validar el flujo completo frente a desarrollar una interfaz a medida. Para precios en tiempo real se eligió \textbf{SearchAPI.io}, que unifica Google Flights y Google Hotels y evita la integración directa con proveedores GDS como Amadeus, reduciendo la complejidad de esta parte del sistema.
